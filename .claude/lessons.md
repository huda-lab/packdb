# Lessons Learned

Corrections and patterns discovered during development. Updated after mistakes to prevent recurrence.

## Rules

- **Grammar files are templates**: Never edit `src_backend_parser_gram.cpp` directly. Edit `.y`/`.yh` files in `third_party/libpg_query/grammar/`, then run `python3 scripts/generate_grammar.py`.
- **Build before testing**: Always `make` (or `make debug`) after grammar or source changes before running tests.

## Gotchas

- **Don't rewrite aggregates to SUM at parse time if the return type matters**: Renaming `AVG(x)` → `SUM(x)` before binding changes the return type from DOUBLE to BIGINT (for integer inputs). DuckDB's comparison binding then casts fractional RHS values (e.g., `0.5`) to BIGINT → `0`, silently breaking constraints. Instead, let the original aggregate flow through binding natively (preserving its return type) and detect it at execution time via `BoundAggregateExpression::function.name`. This applies to any future aggregate rewrite where the original function's return type differs from SUM's. *(Resolved — current AVG implementation rewrites post-binding in `DecideOptimizer::RewriteAvgToSum` with `AVG_REWRITE_TAG` for execution-time scaling. Lesson retained as a general principle for future aggregate rewrites.)*
- **Big-M disjunction sign convention**: For `x ≠ K` via Big-M, BOTH constraints must have z coefficient = `-M` (not `+M`/`-M`). Correct: `x - M*z ≤ K-1` (z=0 → x ≤ K-1) and `x - M*z ≥ K+1-M` (z=1 → x ≥ K+1). Wrong sign (+M on first constraint) makes z=0 trivially satisfy both constraints, leaving x unrestricted.
- **FindDecideVariable is "find any"**: `FindDecideVariable(expr)` recursively returns the first DECIDE variable found in the expression tree. For complex LHS expressions (e.g., `z_0 + z_1`), it finds only the first variable and silently ignores the rest. Use `ExtractLinearTerms` for multi-variable LHS, or check `GetExpressionClass() == BOUND_COLUMN_REF` first.
- **FunctionExpression::is_operator defaults false**: Programmatically-created `FunctionExpression("+", ...)` has `is_operator=false`, unlike parser-created ones where it's `true`. Don't gate logic on `is_operator` for expressions from rewrites.
- **McCormick bilinear rewrite must preserve outer coefficients**: When `FindAndReplaceBilinear` (in `src/optimizer/decide/decide_optimizer.cpp`) replaces a bilinear product like `coeff * b * x` (left-associative parse `(coeff*b)*x`) with the McCormick auxiliary `w`, the `coeff` factor must be folded into the replacement (`coeff * w`) — not silently dropped. The binder inserts implicit CASTs around mixed-type multiplications, so coefficient extraction must unwrap `BoundCastExpression` before pattern-matching the `*` node. Symptom: MINIMIZE objectives produce suboptimal results (solver minimizes `SUM(w)` instead of `SUM(coeff*w)`); MAXIMIZE often masks the bug because the corner solution coincides. See `ExtractMultiplicativeCoefficient` and its call sites in the Bool×Non-Bool branch.
- **Entity-scoped DECIDE variables must register their key columns with the scan**: Columns referenced ONLY by `DECIDE Table.var` (not by SELECT/WHERE/constraints/objective) were silently pruned from the table scan at bind time. The `entity_key_bindings` on `EntityScopeInfo` then resolved to a subset of columns, and the `VarIndexer` grouped rows by whatever column happened to survive (often a WHERE-filter column like `n_regionkey`), collapsing distinct entities. Fix: at bind time in `bind_select_node.cpp`, call `TableBinding::GetColumnBinding(col_idx)` for every entity-key column to force it into `bound_column_ids`; store `BoundColumnRefExpression`s on `LogicalDecide::entity_key_expressions` so the column pruner's `VisitExpression` rebinds them alongside other columns; refresh `entity_scopes[...].entity_key_bindings` from these expressions in `plan_decide.cpp` before resolving physical indices. Symptom: single-table-no-join entity-scoped queries produced suboptimal results (entities collapsed); JOIN queries masked the bug when the join condition happened to force identity columns into the scan.
- **Snapshot aggregate sub-state before rebinding `expr` in optimizer rewrites**: Inside a `VisitExpression`-style rewrite, it is tempting to take a reference like `auto &agg = expr->Cast<BoundAggregateExpression>()` and then reassign `expr = BindAggregateFunction(...)` to swap in the rewritten aggregate. But the reassignment destroys the old `BoundAggregateExpression` that `agg` points into, so any later read of `agg.filter`, `agg.children`, `agg.alias`, etc. is a use-after-free. Always deep-copy or move the needed sub-state into a local `unique_ptr<Expression>` (or equivalent) **before** the `expr = BindAggregateFunction(...)` call, then install it on the new aggregate after. Symptom: `COUNT(x) WHEN cond` silently dropped the WHEN filter because the optimizer was copying a freed `agg.filter` pointer, and the freed memory often still held a plausible-looking expression so there was no crash — just a formulation that ignored the filter. See `RewriteCountInExpression` in `src/optimizer/decide/decide_optimizer.cpp` for the fix pattern (snapshot `filter_copy` up-front, `std::move` onto the new `BoundAggregateExpression`).
- **Read child bindings BEFORE `CreatePlan` on the child, not after**: In `plan_decide.cpp` the code captured `op.children[0]->GetColumnBindings()` *after* calling `CreatePlan(*op.children[0])`. Many logical operators (notably `LogicalProjection`) `std::move` their `expressions` vector into the physical op during CreatePlan, which leaves `GetColumnBindings()` returning an empty vector afterward. `LogicalGet` is immune because its bindings come from `column_ids`, so direct-base-table queries worked while subquery/CTE-wrapped ones silently collapsed to a single entity (`entity_key_physical_indices` stayed empty → zero key columns → every row hashed to the same entity → `keep=0` everywhere). Always snapshot anything you need from a child logical op **before** handing it to `CreatePlan`. Fix: capture `child_bindings` on entry.
