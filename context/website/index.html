Optimization, Native in SQL

PackDB extends DuckDB with declarative optimization.
Select optimal data packages with constraints.
No external tools required.

Get Started View on GitHub
Traditional Approach 
# Traditional: Export, Transform, Solve
import duckdb
import pulp

# Query database
conn = duckdb.connect('db.duckdb')
items = conn.execute("""
    SELECT id, value, weight
    FROM Items
    WHERE category = 'electronics'
""").fetchall()

# Build optimization model
prob = pulp.LpProblem("knapsack", pulp.LpMaximize)
x = [pulp.LpVariable(f"x_{i}", cat='Binary') for i in range(len(items))]

# Add constraints manually
prob += pulp.lpSum(x[i] * items[i][2] for i in range(len(items))) <= 50

# Solve
prob += pulp.lpSum(x[i] * items[i][1] for i in range(len(items)))
prob.solve()

# And then map results back to database...
With PackDB 
-- PackDB: Optimize Directly in SQL
SELECT id, value, weight, x
FROM Items
WHERE category = 'electronics'
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * weight) <= 50
MAXIMIZE SUM(x * value);

Same result, one-tenth the code. No context switching.

The Unbundling Problem

Today's optimization workflows are fragmented across multiple tools and languages.

1. Export Data

  • Query database
  • Materialize to app memory
  • Data staleness risk

2. Transform & Model

  • Write solver code
  • Maintain dual expertise
  • Complex debugging

3. Solve & Map Back

  • Parse solver output
  • Rejoin with database
  • Performance overhead

PackDB collapses this entire workflow into a single SQL query.

Why PackDB?

Bringing the power of optimization solvers directly into your SQL queries.

Native SQL Integration

Express optimization problems as SQL extensions. No context switching between query and optimization languages.

Zero Data Movement

Solve directly on database buffers. Eliminate export/import overhead completely.

Declarative Syntax

Define what to optimize, not how. System handles model formulation automatically.

Built on DuckDB

Leverage columnar storage and vectorized execution. Embedded HiGHS solver for high performance.

Built for Complex Selection Problems

From resource allocation to portfolio optimization, solve real-world problems with SQL.

Resource Allocation

Assign engineers to projects within workload constraints

SELECT engineer_id, project_id, x
FROM Assignments
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * hours) PER engineer_id <= 40
MAXIMIZE SUM(x * skill_match);

Note: PER (grouped constraints) is a planned feature — coming soon.

Portfolio Optimization

Select assets maximizing return under risk limits

SELECT ticker, expected_return, x
FROM Assets
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * risk_score) <= 100
MAXIMIZE SUM(x * expected_return);

Meal Planning

Choose recipes optimizing nutrition within calorie budget

SELECT recipe_name, calories, protein, x
FROM Recipes
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * calories) <= 2000,
    SUM(x * protein) >= 50
MAXIMIZE SUM(x * taste_score);

Inventory Selection

Classic knapsack: maximize value, minimize weight

SELECT item_name, value, weight, x
FROM Inventory
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * weight) <= 50
MAXIMIZE SUM(x * value);

Get Started in Minutes

Three steps to your first optimization query.

1

Install PackDB

git clone https://github.com/huda-lab/packdb.git
cd packdb && git checkout hatim
mkdir build && cd build
cmake .. && make -j4
2

Load Your Data

CREATE TABLE Items (
    id INTEGER,
    value INTEGER,
    weight INTEGER
);

INSERT INTO Items VALUES
    (1, 100, 20),
    (2, 60, 10),
    (3, 120, 30);
3

Write a DECIDE Query

SELECT id, value, weight, x
FROM Items
DECIDE x
SUCH THAT
    x IS BINARY,
    SUM(x * weight) <= 50
MAXIMIZE SUM(x * value);
