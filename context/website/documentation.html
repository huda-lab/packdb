Syntax Overview

PackDB extends standard SQL with a DECIDE clause that appears after WHERE and before the final projection.

Complete Syntax

SQL 
SELECT select_list
FROM table_expression
[WHERE filter_conditions]
DECIDE variable_list
SUCH THAT
    constraint_expression
    [AND constraint_expression2 ...]
[MAXIMIZE | MINIMIZE] objective_expression

Execution Order

  1. 1
    FROM + WHERE — Standard SQL filtering produces candidate rows
  2. 2
    DECIDE — Creates decision variables (one per candidate row)
  3. 3
    SUCH THAT — Defines optimization constraints
  4. 4
    MAXIMIZE/MINIMIZE — Specifies objective function
  5. 5
    SELECT — Projects results including decision variable values

Decision Variables

Variables declared in the DECIDE clause represent choices to be made for each candidate row.

Syntax

SQL 
DECIDE variable_name [, variable_name2 ...]

Variable Types

Binary Variables

SQL 
DECIDE x
SUCH THAT x IS BINARY
  • Domain: {0, 1}
  • Use case: Selection decisions ("include this row or not")
  • Internally: Integer variable with bounds [0, 1]

Integer Variables

SQL 
DECIDE x
SUCH THAT x IS INTEGER
  • Domain: Non-negative integers only: {0, 1, 2, 3, ...}
  • Use case: Quantity decisions ("how many of this item")
  • Bounds: x >= 0 is automatically applied
  • Note: Negative values are not allowed by default

Variable Scope

Decision variables are available in:

  • SUCH THAT clause (constraints)
  • MAXIMIZE/MINIMIZE clause (objective)
  • SELECT clause (output projection)
SQL 
-- Valid: Using x in all three places
SELECT id, price, x, (x * price) as cost
FROM Products
DECIDE x
SUCH THAT x IS BINARY
MAXIMIZE SUM(x * profit);

Multiple Variables

SQL 
-- Two decision variables per row
DECIDE x, y
SUCH THAT
    x IS BINARY
    AND y IS INTEGER
    AND x + y <= 10

Important Notes

  • Each row gets its own instance of each variable
  • Variables exist only within the query scope
  • No support for real (continuous) decision variables

Constraints

Constraints in the SUCH THAT clause define feasible solutions.

Constraint Types

Row-Level Constraints

Applied independently to each row.

SQL 
-- Each x must be between 0 and 5
SUCH THAT
    x >= 0
    AND x <= 5

Aggregate Constraints

Applied to the collection of selected rows.

SQL 
-- Total cost across all selected rows
SUCH THAT
    SUM(x * price) <= 1000

Supported Operators

Operator Meaning Example
= Equality x = 1
<> Not equal (BINARY variables only) x <> 0
< Less than x < 10
<= Less than or equal SUM(x * weight) <= 50
> Greater than x > 0
>= Greater than or equal SUM(x) >= 3

Supported Constructs

BETWEEN

SQL 
SUCH THAT x BETWEEN 1 AND 10

-- Equivalent to:
SUCH THAT x >= 1 AND x <= 10

IN

SQL 
SUCH THAT x IN (1, 2, 3)

-- IN is handled as a native construct (not via OR expansion)

Scalar Subqueries

SQL 
-- Bound computed from another table
SUCH THAT
    SUM(x) <= (SELECT COUNT(*) FROM Drivers)

Important: Only uncorrelated scalar subqueries are supported. The subquery cannot reference columns from the outer query.

SUM(x * price) <= (SELECT AVG(budget) FROM Departments)
x <= (SELECT budget FROM Depts WHERE Depts.id = Items.dept_id)

Linearity Requirement

All constraints must be linear in decision variables

Valid Linear Expressions

SQL 
-- Constant coefficient
x * 5

-- Table column coefficient
x * price

-- Sum of linear terms
x + y + z

-- Complex but linear
SUM(x * price + y * cost - z * discount)

Invalid Non-Linear Expressions

SQL 
-- Quadratic term
x * x

-- Variable interaction
x * y

-- Non-linear function
SIN(x)

-- Division by variable
cost / x

Constraint Examples

SQL 
-- Simple bounds
SUCH THAT
    x >= 0
    AND x <= 1

-- Multiple constraints
SUCH THAT
    x IS BINARY
    AND SUM(x * weight) <= 100
    AND SUM(x * volume) <= 50
    AND SUM(x) >= 5

-- Complex coefficients
SUCH THAT
    SUM(x * (price + tax) * quantity) <= budget

-- Subquery bound
SUCH THAT
    SUM(x) <= (SELECT capacity FROM Config WHERE name = 'max_items')

Objective Functions

The objective function specifies what to optimize.

Syntax

SQL 
MAXIMIZE aggregate_expression
-- or
MINIMIZE aggregate_expression

Requirements

  1. Must use SUM() aggregation
  2. Must involve at least one decision variable
  3. Must be linear

Examples

SQL — Maximize Total Value 
MAXIMIZE SUM(x * value)
SQL — Minimize Total Cost 
MINIMIZE SUM(x * cost)
SQL — Complex Coefficients 
MAXIMIZE SUM(x * (revenue - cost) * discount_factor)
SQL — Multiple Variables 
MAXIMIZE SUM(x * profit_a + y * profit_b)

Unsupported Aggregations

SQL — Not Supported 
-- COUNT not supported
MAXIMIZE COUNT(x)

-- Workaround: Use SUM with binary variable
DECIDE x
SUCH THAT x IS BINARY
MAXIMIZE SUM(x)

-- AVG not supported (non-linear ratio)
MAXIMIZE AVG(x * value)

-- MAX/MIN not supported in objective
MAXIMIZE MAX(x * value)

Note: Constant terms in the objective are ignored, as they do not affect the optimal solution.

Solver Outcomes

PackDB only returns results when the optimization problem is successfully solved to optimality. Here's what happens with different solver outcomes:

Optimal

The solver found an optimal solution.

Result: Query returns rows with their optimal decision variable values.

Infeasible

No valid solution exists—the constraints cannot all be satisfied simultaneously.

Result: Returns an internal error.

Unbounded

The objective can grow infinitely—no finite optimum exists.

Result: Returns an internal error.

Note: If your query returns an error, check that your constraints are not contradictory (infeasible) and that your objective is properly bounded by the constraints (unbounded).

Features & Limitations

Fully Supported

Feature Status Example
Binary variables Full x IS BINARY
Integer variables Full x IS INTEGER
Linear constraints Full SUM(x * col) <= bound
Aggregate constraints Full SUM(x * weight) <= 50
Row-level constraints Full x >= 0 AND x <= 10
Scalar subqueries Uncorrelated only ... <= (SELECT ...)
BETWEEN Full x BETWEEN 1 AND 10
IN clauses Full x IN (1, 2, 3)
WHERE filtering Full Standard SQL
Multiple variables Full DECIDE x, y, z

Technical Constraints

Linearity Requirement

PackDB currently supports only Linear Programming (LP) and Integer Linear Programming (ILP).

Cannot express:

  • Quadratic objectives (e.g., x * x)
  • Variable interactions (e.g., x * y)
  • Non-linear functions (e.g., LOG(x), EXP(x))

Subquery Restrictions

Only uncorrelated scalar subqueries are supported.

SUM(x) <= (SELECT AVG(capacity) FROM Resources)
x <= (SELECT budget FROM Depts WHERE Depts.id = Items.dept_id)

Future: The roadmap includes correlated subquery support via query unnesting.

Scalability

The current prototype uses an exact ILP solver (HiGHS).

Performance Profile
Fast
< 1,000 rows
Acceptable
1K - 10K rows
Slow
> 10,000 rows

Why? Integer programming is NP-Hard. Exact solvers guarantee optimality but cannot scale to millions of rows.

Future: The roadmap includes support for the Sketch-Refine approximation algorithm, which will enable million-row datasets with bounded approximation error.

Memory Requirements

The PhysicalDecide operator materializes all candidate rows in memory.

Impact: Very large candidate sets may exceed available RAM.

Mitigation: Use WHERE clauses aggressively to filter candidates before optimization.

Single Decision Context

JOINs work fine with PackDB—decision variables operate on the result of the WHERE/HAVING clauses (the joined rows).

SQL — Supported 
-- JOINs work: DECIDE operates on the joined result set
SELECT o.id, p.name, x
FROM Orders o
JOIN Products p ON o.product_id = p.id
DECIDE x
SUCH THAT x IS BINARY
MAXIMIZE x * p.profit

What is not supported is declaring separate decision variables tied to different source tables:

SQL — Not Supported 
-- Not Supported: per-table variable declarations
DECIDE x ON orders, y ON customers  -- Cannot bind variables to specific tables

All decision variables share a single context: the rows produced by the WHERE/HAVING clauses.

Best Practices

1 Filter Aggressively with WHERE

The optimization problem size grows with the number of candidate rows. Use WHERE to reduce the input.

Slow 

SELECT *
FROM Products
DECIDE x
SUCH THAT x IS BINARY
MAXIMIZE SUM(x * value);

Better 

SELECT *
FROM Products
WHERE category = 'electronics'
  AND price > 100
  AND in_stock = true
DECIDE x
SUCH THAT x IS BINARY
MAXIMIZE SUM(x * value);

2 Use Binary Variables When Possible

Binary variables ({0, 1}) solve faster than integer variables.

SQL 
-- If you only need "select or not", use BINARY
DECIDE x
SUCH THAT x IS BINARY

-- Only use INTEGER when you need quantities
DECIDE quantity
SUCH THAT quantity IS INTEGER

3 Start Simple, Debug Incrementally

Build your query incrementally:

  1. Start with a single constraint
  2. Verify the solution makes sense
  3. Add more constraints one at a time
  4. Test with small datasets first

If your query returns an error, the constraints might be infeasible. Try relaxing them:

SQL — Debug by relaxing constraints 
SUCH THAT
    x IS BINARY
    AND SUM(x * weight) <= 50  -- Try increasing this
    AND SUM(x) >= 10           -- Maybe this is too strict?

Tip: Run the query without the DECIDE clause first to see how many candidate rows exist.
