# 004 — CSR Constraint Storage + Gurobi Bulk addconstrs + Early Backend Reject

**Commit**: `efee33d163` (`decide: CSR constraint storage + Gurobi bulk addconstrs + early backend reject`)
**Baseline**: `d711f1b6fb`
**Benchmark JSONs**: `benchmark/decide/results/efee33d163.json` vs `benchmark/decide/results/d711f1b6fb.json`
**Suite**: `benchmark/decide/run_benchmarks.py`, sizes `medium` + `large`, 3 iterations each.

## Changes

Solver-handoff cluster (Option A from `todo.md`). Three connected edits, unified by storing the linear constraint matrix once in CSR form on `SolverModel`.

1. **CSR storage in `SolverModel`.** Replaced `vector<ModelConstraint> constraints` with `row_start` / `col_index` / `value` / `sense` / `rhs` (CSR row-major). `ModelConstraint` is now a builder-local scratch struct in an anonymous namespace inside `ilp_model_builder.cpp`; `PushNormalizedConstraint` flattens it into the model's CSR arrays on emission and appends the next `row_start` sentinel. Reservation block, raw-global-constraint append, and sanity loop are all CSR-aware.

2. **HiGHS direct CSR ingest.** `deterministic_naive.cpp` no longer builds COO triplets and converts to CSR — `row_lower` / `row_upper` are derived from `model.sense` / `rhs` in one branchy loop and `model.row_start` / `col_index` / `value` are assigned straight into `HighsLp::a_matrix_`.

3. **Gurobi bulk `GRBaddconstrs`.** Added `addconstrs` to the loader as an optional symbol (mirrors the `addqconstr` pattern). `GurobiSolver::Solve` issues a single `GRBaddconstrs` over the CSR storage when the symbol is loaded, with a per-row `GRBaddconstr` fallback otherwise.

4. **Backend selection + early validation.** `SolveModel()` now inspects `SolverInput` upfront for quadratic constraints, non-convex objective (including bilinear), and MIQP, picks the backend, and throws the existing HiGHS-incompatibility errors **before** `SolverModel::Build()` runs. Saves the Q/QC outer-product expansion when HiGHS is pinned on an incompatible model. The in-`Solve` HiGHS rejections in `deterministic_naive.cpp` are kept as defense-in-depth.

## Hypothesis

- Removing the COO→CSR step in HiGHS should shrink `solver_ms` modestly (one matrix copy + one prefix-sum pass eliminated).
- `GRBaddconstrs` collapses N C-API trips into one — proportional savings only when N is large.
- Early backend reject affects correctness/latency only when HiGHS is pinned on a Gurobi-only model (no benchmark hits this path).

## Why the wins didn't surface here

- TPC-H DECIDE queries used here have **at most 9 constraints** (Q8 large; most have 1–7). Collapsing 9 `addconstr` C-API calls into one `addconstrs` call is invisible against a multi-second Gurobi solve.
- Every bench query runs on Gurobi (where Build's COO storage was already a single-pass loop). The HiGHS COO→CSR work that this commit deletes never executes on this suite.
- The early backend reject only fires when HiGHS is forced on a Gurobi-only model — none of the bench queries trigger that path.
- Solver wall-time variance (Gurobi branch-and-cut) is several seconds at the large scale, swamping any handoff-level delta. Q9 in particular ranges ~13–18s at medium and ~19–27s at large; Q9 large even had one run fail outright on the new build, which the median absorbs.

## When this change *should* show up in numbers

A workload with many constraints (hundreds-to-thousands of MIN/MAX/ABS auxiliary rows, per-row constraints over `num_rows = 1e6`) on the **HiGHS** backend would exercise both wins simultaneously: the COO build elimination scales with `total_nnz`, and the `addconstrs` collapse scales with row count. The current TPC-H suite is the wrong shape to measure it; future regression queries with thousands of per-row linking rows would.
