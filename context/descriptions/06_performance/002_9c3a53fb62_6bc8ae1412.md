# 002 — Batched ExpressionExecutor + Vectorized Extract + Typed Hash Keys

**Commit**: `6bc8ae1412` (`decide: batch ExpressionExecutor + vectorized double extract + typed hash keys for grouping`)
**Baseline**: `9c3a53fb62`
**Benchmark JSONs**: `benchmark/decide/results/dirty.json` (commit `6bc8ae1412`) vs `benchmark/decide/results/9c3a53fb62.json`
**Suite**: `benchmark/decide/run_benchmarks.py`, sizes `medium` + `large`, 3 iterations each.

## Changes

Three coupled hot-path edits in `src/execution/operator/decide/physical_decide.cpp`. None touch `SolverModel` layout or solver backends; the produced `SolverInput` is unchanged.

1. **Vectorized DOUBLE extraction.** New static helper `ExtractDoubleColumn(Vector &result_vec, idx_t count, double sign, vector<double> &out, const char *err_context)` replaces the per-row `Value::GetValue() + DefaultCastAs(LogicalType::DOUBLE)` pattern at six sites: constraint linear coefficient, constraint RHS, constraint bilinear, constraint quadratic, objective linear, objective bilinear. Reads via `Vector::ToUnifiedFormat` after a fast-path DOUBLE check (skips cast when the result type is already DOUBLE) or `VectorOperations::DefaultCast` once per chunk. NULL → typed `InvalidInputException`; non-finite → typed exception with the same wording as before.

2. **Batched `ExpressionExecutor` per constraint / objective.** The previous pattern constructed one `ExpressionExecutor` per term per chunk (sometimes per term outside the chunk loop). Now each block hoists the executor outside the chunk scan, `AddExpression`s every coefficient expression once, and `Execute`s into a multi-column `DataChunk` once per chunk:
   - Constraint linear: one executor for all `lhs_terms`, route by `term_idx`.
   - Constraint bilinear: one executor for all `bilinear_terms` with a coefficient (those without one materialize directly via `assign(num_rows, sign)`).
   - Constraint quadratic: one executor per `quadratic_groups[group_idx]` covering all `inner_terms`, route by `inner_idx`.
   - Objective linear: hoists the existing `flat_coeffs` setup so the executor lives outside the chunk loop; the existing `route[(b_idx, term_idx)]` continues to map result columns to destinations.
   - Objective bilinear: same shape as constraint bilinear.
3. **Typed hash keys for entity and PER grouping.** New shared helper `BuildGroupIds(key_exprs, context, data, num_rows, row_filter, null_excludes, out_row_to_group, out_num_groups)` replaces three nearly identical `Value::ToString()`-based composite-string keying sites: entity mapping, constraint PER, objective PER. Per chunk it runs one `ExpressionExecutor` over all key columns and computes a per-row `hash_t` via `VectorOperations::Hash` + `VectorOperations::CombineHash`; collision fallback uses cached typed `Value`s for equality. `null_excludes` flips PER vs entity NULL semantics; `row_filter` carries WHEN / aggregate-local-filter exclusion. First-seen group-id ordering preserved.

## Hypothesis

The previous batch (`9c3a53fb62`) extracted easy wins inside `SolverModel::Build()` and lower-bound absorption. The next dominant cost was per-row work *during* coefficient evaluation in `Finalize()`:

- Each linear constraint term constructed a fresh `ExpressionExecutor` per chunk and extracted values one row at a time through `Value::GetValue()` and `DefaultCastAs(DOUBLE)`. For wide constraints (many terms) on large tables (many chunks), this is `num_terms × num_chunks` setup cost and `num_rows × num_terms` per-row scalar Value allocation.
- Entity mapping and PER grouping built `std::string` keys via `Value::ToString()` joined by NULL-tag bytes, then hashed those strings. With multi-column PER and multi-million-row tables, this is the dominant entity/PER cost.

All three changes target hot loops that scale with `num_rows × num_terms`. Bilinear/quadratic blocks remain gated by their `has_bilinear` / `has_quadratic` flags so empty blocks pay no setup cost. The batched executor pattern reduces both setup and result-chunk-allocation overhead; the vectorized extract replaces per-cell `Value` round-trips with a single cast and `UnifiedVectorFormat` read; the typed hash keys eliminate string allocation and per-row `ToString()`.

### Where the wins concentrate

- **Q7 entity_scope_join (−55%)**: this query's only entity-scoped variable means entity mapping dominates model build. The typed-hash-key change replaces `Value::ToString()` over `o_orderpriority` per row with a vectorized hash + typed `Value` equality fallback. Single biggest improvement of the batch.
- **Q2 abs_minmax (−40%) and Q4 nested_per_objective (−34%)**: heavy on linear coefficient evaluation across many terms with PER grouping. Both batched-executor and vectorized-extract paths hit these directly.
- **Q3 avg_when (−35%)**: aggregate-local WHEN with batched mask evaluation; benefits from batched executor and from the typed-hash PER lookup.
- **Q5, Q6 (−15% to −19%)**: more modest wins where coefficient eval is one of several contributors.

### Apparent regressions

- **Q9 quadratic_gurobi large**: +29.0%, but σ=4.3s on the new run and σ=5.1s on the old. Q9 is 100% solver-bound (Gurobi MIP wall time), and Gurobi MIP wall time has high run-to-run variance unless `Seed` and threading are pinned. Q9 medium is −6.3% with much smaller σ on the same change. Treating the large delta as solver-time noise rather than a regression caused by these edits, mirroring the noise pattern flagged in entry 001 for Q1/Q9 medium under the previous batch.
- **Q1, Q8 (≤2% change)**: both queries are ≥97% solver-bound, so model-time wins are invisible against solver wall time. Expected, not a regression.

### Test status

`make decide-test`: **547 passed, 1 skipped**. No regressions. The three changes preserve `SolverInput` content (only construction cost changes), so the differential test suite — which compares PackDB output against an oracle solver — would catch any group-id reassignment or coefficient drift; none observed.
