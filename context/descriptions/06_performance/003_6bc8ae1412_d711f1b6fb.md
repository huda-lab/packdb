# 003 — Single VarIndexer + Moved Raw Constraints + SparseCoeffAccumulator

**Commit**: `d711f1b6fb` (`decide: build VarIndexer once, move raw constraints, sparse accumulator for grouped LHS`)
**Baseline**: `6bc8ae1412`
**Benchmark JSONs**: `benchmark/decide/results/d711f1b6fb.json` vs `benchmark/decide/results/dirty.json` (commit `6bc8ae1412`, only doc/markdown files dirty in worktree — no perf-affecting code)
**Suite**: `benchmark/decide/run_benchmarks.py`, sizes `medium` + `large`, 3 iterations each.

## Changes

Three contained edits to the `SolverInput` → `SolverModel` handoff. None change `SolverInput` content; only construction and accumulation costs change.

1. **`VarIndexer` built once per finalize.** Previously built three times: a `pre_indexer` (BuildRef) for deferred-NE / MIN-MAX expansion in `physical_decide.cpp`, a fresh `BuildRef` inside `SolverModel::Build()`, and an owning `Build()` for `gstate.var_indexer` at the end of finalize. Now a single owning `var_indexer = VarIndexer::Build(solver_input)` is created early in `Finalize()`, used everywhere downstream (replacing `pre_indexer`), threaded through `SolveModel()` and `SolverModel::Build()` as a `const VarIndexer &indexer` parameter, and `std::move`d onto `gstate.var_indexer` after the solve. `total_vars` is refreshed (`global_block_start + num_global_vars`) just before the solve to account for global aux vars added during finalization. The non-owning `BuildRef()` form is retained in the header but has no production callers now.

2. **Raw global constraints moved into `SolverModel`.** `SolverModel::Build` now takes `SolverInput &input` (non-const) instead of `const SolverInput &`, and `std::move`s the `indices` and `coefficients` vectors out of each `RawConstraint` into the new `ModelConstraint` instead of deep-copying. `SolveModel()` is the only caller and never reads `input.global_constraints` after `Build()` returns.

3. **`SparseCoeffAccumulator` replaces per-group `unordered_map<int, double>`.** New struct in `src/include/duckdb/packdb/ilp_model.hpp`, definitions in `ilp_model_builder.cpp`. Two strategies, picked once per constraint and reused across groups within the constraint:
   - **Dense**: `vector<double>` indexed by flat var idx, with a `touched` list of written indices so `Flush` zeros only those cells (O(touched), not O(total_vars)). Used when the decide-variable index span (`var_indexer.global_block_start`, tighter than `total_vars` because decide vars never live in the global block) fits a 1M-slot / 8MB cap.
   - **Sparse**: append-only `(idx, coeff)` pair list, sorted by index and merged on `Flush`. Fallback when the decide-var span exceeds the cap.

   Both call sites adopted: the grouped-constraint slow path in `ilp_model_builder.cpp` (`!CanUseRowScopedFastPath`), and the deferred aggregate `<>` expansion in `physical_decide.cpp` where the deduped LHS feeds two raw constraints (`SUM - M·z ≤ K-1` and `SUM - M·z ≥ K+1-M`) — `Flush` once into a local snapshot, then copy/move into `rc1`/`rc2`. The pre-existing `std::abs(coeff) < 1e-15` skip in the deferred-NE inner loop is preserved to avoid any semantic drift.

## Hypothesis

Each item targets a different small-but-hot inefficiency in the model-construction stage, not the solver:

- **Item 1**: `VarIndexer::Build()` and `BuildRef()` both iterate `num_decide_vars` to populate scope vectors and entity-mapping pointers. Three builds = three iterations + one entity-mappings copy. Going to one build saves the iteration twice and clarifies the lifecycle.
- **Item 2**: `RawConstraint` carries two `vector` members; deep-copying them into `ModelConstraint` is `O(num_global_constraints × avg_terms)` heap copies. For queries that produce many global aux constraints (deferred-NE expansion, MIN/MAX linking), this is a measurable allocation churn.
- **Item 3**: `unordered_map<int, double>` allocated/destroyed per group has malloc/free + bucket-array alloc cost, and rehashes during growth. A reused dense vector + touched list amortizes one allocation across all groups in a constraint, with O(1) inserts and O(touched) flush. Sorted pairs handle the rare giant-`total_vars` case without unbounded scratch memory.

The biggest impact should land on queries that exercise the deferred-NE path or the entity-scoped accumulator slow path — i.e., aggregate `<>` constraints (Q6) and grouped/entity-scoped accumulating constraints. Queries that are pure row-scoped fast-path (Q1, Q5, Q8, Q9) should be near-flat on model time, since they never touch the changed accumulator.

### Where the wins concentrate

- **Q6 domain_ne_between (−25% / −26% on model stage)** is the headline. Q6 is the only query in the suite that exercises the deferred aggregate `<>` path heavily — exactly where `SparseCoeffAccumulator` *and* the single-`var_indexer` change both land on the same code. Both contribute: per-group hash-map allocation gone (item 3) and three `VarIndexer` setups collapsed into one (item 1). Wall is unchanged because the model stage is sub-1% of Q6's total wall — solver dominates — but the model-stage delta is real, well outside any per-run noise we see on this query.

### Near-flat queries

- Q2, Q3, Q4, Q5, Q8: model-stage changes are within ±3%, consistent with run-to-run noise. These queries' grouped paths often hit the row-scoped fast path or use small enough constraint counts that the single `VarIndexer` save (microseconds) and the `RawConstraint` move (no/few raw constraints) don't surface.

### Apparent regressions on model stage

- **Q1 medium +10.0%, large +3.1%**: absolute Δ is +0.6 ms / +0.4 ms. Q1's model stage is sub-7 ms; a 0.5 ms shift sits inside the per-iteration noise band we see on tiny stages. No code path was added in Q1's flow.
- **Q7 entity_scope_join +1.0% / +0.7%**: small (+2–4 ms) on a 250–515 ms model stage. Q7 hits the entity-scoped accumulator slow path, so the change *should* affect it; the marginal regression may be the dense-`vector<double>` zero-init the first time `BeginDense` runs, before `touched`-list reuse amortizes across groups. Within noise, but worth re-checking if a follow-up touches this path.
- **Q9 quadratic_gurobi +4.5% / +2.1%**: absolute +0.3 ms on a 6–14 ms stage. Q9 is 100% solver-bound at wall level; the model-stage delta is irrelevant to total time and inside noise.

### Wall-clock noise notes

- **Q9 quadratic_gurobi large −12.6%, medium +25.8%**: pure Gurobi MIP variance. Q9's model stage barely moved (≤+0.3 ms); the solver swung 13–25%. Mirrors the Q9 noise pattern flagged in entries 001 and 002. Not attributable to this change in either direction.
- **Q1 large −4.3% wall**: solver, not us — model stage is +0.4 ms (well under 0.01% of wall).
